## 1. Назначение и область применения
Документ задает обязательные правила оформления и проверки кода в Prompt Assistant.

Основная часть фиксирует обязательные нормы. Практический контроль вынесен в приложение.

## 2. Единые требования ко всем языкам
- Все комментарии, docstrings и JSDoc в проекте пишутся **только на русском языке**.
- Комментарии описывают неочевидную логику, а не очевидные операции.
- Любое отклонение от стандартов должно быть обосновано в PR.

## 3. Стандарты Python
### 3.1 Базовые нормы
- Обязательные основы: **PEP 8** (стиль), **PEP 257** (docstrings).
- Формат docstrings: **Google Style**.
- Публичные сущности документируются: модули, классы, функции, методы.

### 3.2 Докстринги и комментарии
- Каждый Python-модуль начинается с docstring.
- Для публичных функций/методов обязательны блоки `Args`, `Returns`, `Raises` (если применимо).

Пример:
```python
def calculate_price(amount: int, discount: float = 0.0) -> float:
    """Рассчитывает итоговую цену с учетом скидки.

    Args:
        amount: Базовая сумма в рублях.
        discount: Процент скидки в диапазоне 0..100.

    Returns:
        Итоговая сумма после применения скидки.

    Raises:
        ValueError: Если скидка вне диапазона 0..100.
    """
    if not 0 <= discount <= 100:
        raise ValueError("Скидка должна быть в диапазоне 0..100")
    return amount * (1 - discount / 100)
```

### 3.3 Импорты и именование
- Порядок импортов: стандартная библиотека → сторонние пакеты → локальные `pa_core.*`.
- Запрещены `from module import *` (кроме явных модулей с `__all__`) и относительные импорты вида `.models`.
- Именование по PEP 8: `lowercase_with_underscores` для модулей/функций/переменных, `CapWords` для классов, `ALL_CAPS` для констант.

### 3.4 Типизация
- Типизация обязательна для публичных функций/методов и значимой внутренней логики.
- `Any` запрещен; исключение допускается только с явным комментарием-обоснованием.
- API-контракты и валидация: Pydantic. Доступ к БД: SQLAlchemy + Alembic.
- Режим проверки типов: `mypy --strict`.

## 4. Стандарты TypeScript / React
### 4.1 Типизация
- В `tsconfig.json` обязателен `"strict": true`.
- `any` запрещен. Если тип неизвестен, использовать `unknown` с последующей проверкой.
- Пропсы компонентов типизируются через интерфейс `Props`.
- Типы API-ответов описываются явно; переиспользуемые типы выносятся в `src/types/`.

### 4.2 Документация и комментарии
- Для публичных функций и компонентов используется JSDoc на русском языке.
- Для сложных кастомных хуков JSDoc обязателен.

### 4.3 Структура и импорты
- Базовая структура фронтенда: `frontend/src/{admin,components,pages,hooks,types,utils}`.
- Один компонент — один файл (исключения только для мелких вспомогательных компонентов).
- Порядок импортов: внешние библиотеки → внутренние модули → стили.
- Используются алиасы импортов (например, `@/components/...`) при наличии настройки в `vite.config.ts` и `tsconfig.json`.

### 4.4 React-правила
- Используются функциональные компоненты.
- Хуки вызываются только на верхнем уровне компонента.
- В JSX не создавать анонимные функции без необходимости в горячих участках рендера.
- Для объединения классов использовать `clsx`/`cn`.
- Для работы с серверными данными обязателен React Query (TanStack Query).

### 4.5 Ошибки и логирование
- Ошибки в асинхронных операциях обрабатываются через `try/catch` или `.catch()`.
- Для ошибок рендера использовать Error Boundary.
- Прод-ошибки отправляются через Sentry-совместимый клиент (GlitchTip).

## 5. Инструменты автоматического контроля
Обязательные проверки в CI:
- Python: `ruff`, `mypy`, тесты;
- Frontend: `tsc`, `eslint`, `prettier`, тесты;
- Локально перед push: pre-commit/lint-staged.

Рекомендуемая база:
- Python: `py311`, line length `88`, форматирование через Ruff formatter;
- TS/React: ESLint (TypeScript + React hooks + import order), Prettier.

Нарушение обязательных проверок блокирует слияние PR.

## 6. Минимальные требования безопасности в коде
- Не хардкодить секреты, токены, пароли, ключи.
- Не использовать строковую конкатенацию SQL (только параметризованные запросы).
- Запрещено `except Exception: pass`.
- Пароли хешировать через `bcrypt` (cost factor 12).
- Не использовать `dangerouslySetInnerHTML` без обязательной санитизации.

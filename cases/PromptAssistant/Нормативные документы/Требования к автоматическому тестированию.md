## 1. Назначение и область применения
Документ задаёт обязательные требования к автоматическому тестированию Prompt Assistant: состав тестов, уровни покрытия, безопасность тестового процесса и правила встраивания проверок в CI/CD.

Требования распространяются на:
- backend и frontend код;
- API и интеграции;
- пайплайн генерации и связанные сервисы;
- релизный контур CI/CD.

## 2. Общие принципы

1. **Тесты обязательны для всего критического функционала** и выполняются автоматически перед каждым развёртыванием (CI/CD).
2. **Изоляция внешних зависимостей** – все вызовы LLM, платёжных систем, Telegram, email замоканы; реальные вызовы допускаются только на специальном тестовом стенде.
3. **Тестовая база данных** – отдельный экземпляр PostgreSQL (с pgvector), пересоздаваемый перед каждым прогоном.
4. **Воспроизводимость** – тесты не должны зависеть от времени суток, внешних данных, случайности.
5. **Покрытие кода** (ориентир):
   - Критический код (auth, billing, generation pipeline) – не менее 85%.
   - Остальной код – не менее 70%.
6. **Документирование** – каждый тестовый файл содержит краткое описание проверяемого сценария; имена тестов отражают суть проверки.

## 3. Уровни тестирования

### 3.1 Модульные тесты
- **Область:** изолированные функции, утилиты, без обращения к БД и сетям.
- **Инструменты:** `pytest`, `pytest-asyncio`, `unittest.mock`.
- **Примеры:** хеширование паролей, вычисление стоимости LLM, нормализация URL, валидация Pydantic-схем, работа с кэшем лимитов, структурированное логирование.

### 3.2 Интеграционные тесты
- **Область:** взаимодействие с БД, кэшем, очередями, внешними API через моки; тестирование API-эндпоинтов.
- **Инструменты:** `pytest` + `httpx.AsyncClient`, `freezegun`, `faker`, `pytest-docker` для развёртывания тестовой инфраструктуры.
- **Покрытие:**
  - CRUD промптов, цепочек, пользователей.
  - Логика тарифов и лимитов (списания, блокировки).
  - Учёт затрат LLM (таблица `llm_costs`).
  - Отправка email и Telegram-уведомлений (моки).
  - Webhook-обработчики платежей (имитация YooKassa/Robokassa).
  - Проверка прав доступа (роли, изоляция данных пользователей).

### 3.3 End-to-End (E2E) тесты
- **Область:** полные сценарии пользователя (регистрация → генерация → сохранение → оплата).
- **Инструменты:** `Playwright` (Chromium) для UI; отдельный тестовый стенд с поднятым приложением и всеми зависимостями.
- **Покрытие:**
  - Воронка активации.
  - Работа библиотеки (поиск, редактирование, импорт/экспорт).
  - Админ-панель (управление пользователями, тарифами, интеграциями).

### 3.4 Тесты AI-пайплайна (специализированные)
- **Область:** проверка 7-шаговой генерации без вызова реальной LLM (используются предопределённые ответы из эталонного датасета).
- **Инструменты:** `pytest` + мок `call_llm_chat`.
- **Эталонные данные:** `tests/evaluation/golden/` – минимум 5 кейсов на модальность (text-craft и др.). Каждый кейс содержит входной запрос, ожидаемый QC-скор и ключевые слова.
- **Проверки:**
  - Корректность всех шагов пайплайна.
  - Обработка уточняющих вопросов (переход в диалог).
  - Вычисление оценки качества (QC), аномалии (draft > final).
  - Падение при ошибке LLM.

### 3.5 Нагрузочное тестирование
- **Цель:** проверить производительность системы под нагрузкой и выявить узкие места.
- **Инструменты:** `locust` или `k6`.
- **Сценарии:**
  - Генерация промптов (`/generate`) – 10–50 параллельных пользователей.
  - Поиск по библиотеке (с семантическим фильтром).
  - Импорт файлов.
  - Аутентификация (логин/регистрация).
- **Критерии:**
  - Время ответа p95 < 5 секунд (для генерации) / < 1 секунда (для остальных эндпоинтов).
  - Процент ошибок (5xx) < 1% при пиковой нагрузке.
  - Отсутствие утечек памяти, стабильность очередей.
- **Запуск:** перед major-релизами или при подозрении на деградацию; отчёты сохраняются в артефактах.

## 4. Тестирование безопасности

### 4.1 Статический анализ (SAST)
- **Инструменты:** `bandit` (Python), `eslint-plugin-security` (фронтенд), `trivy` (образы).
- **Запуск:** при каждом коммите или ежедневно в CI.
- **Проверки:**
  - Отсутствие хардкода секретов.
  - Безопасное использование `eval`, `exec`, динамической загрузки кода.
  - Параметризованные SQL-запросы (исключение инъекций).
  - Опасные конструкции в шаблонах (`dangerouslySetInnerHTML` и т.п.).
  - Устаревшие/уязвимые библиотеки.

### 4.2 Сканирование зависимостей
- **Инструменты:** `pip-audit`, `npm audit`, `trivy`.
- **Запуск:** еженедельно или при обновлении зависимостей.
- **Правила:** критическая уязвимость (CVSS ≥ 7.0) устраняется в течение 48 часов; остальные – в рамках планового обновления.

### 4.3 Динамическое тестирование (DAST)
- **Инструмент:** `OWASP ZAP`.
- **Сценарии:**
  - Сканирование всех публичных эндпоинтов (регистрация, логин, генерация, API) после каждого деплоя.
  - Ежемесячное полное сканирование.
  - Проверка, что административные эндпоинты недоступны извне (403/404).
  - Проверка защиты от CSRF (наличие токенов, проверка Origin/Referer).

### 4.4 Тесты авторизации и прав доступа
- Интеграционные тесты должны подтверждать:
  - Пользователь не может просматривать/редактировать чужие промпты.
  - Обычный пользователь не имеет доступа к эндпоинтам админ-панели (403).
  - При недействительном/просроченном JWT возвращается 401.
  - Невозможность обойти аутентификацию прямыми ссылками на защищённые ресурсы.
- Каждый маршрут с ролевыми ограничениями тестируется с анонимом, обычным пользователем и администратором.

### 4.5 Валидация входных данных
- Все поля от клиента проходят валидацию по длине, формату, допустимым символам.
- Спецсимволы экранируются или отвергаются в зависимости от контекста.
- Загрузка файлов ограничена по расширениям и MIME-типам; проверяется, что файл не содержит исполняемый код.

### 4.6 Безопасность тестовой среды
- Тесты не используют реальные секреты; внешние вызовы замоканы или направлены на тестовые стенды.
- Тестовая БД пересоздаётся перед каждым прогоном и не содержит реальных пользовательских данных.
- Все тесты выполняются в изолированных контейнерах CI без доступа к продакшн-ресурсам.

### 4.7 Регулярный пентест
- Не реже одного раза в год – внешний пентест с привлечением специалистов.
- Критические уязвимости устраняются в приоритетном порядке.

## 7. CI/CD интеграция

- **При каждом коммите в репозиторий:** запуск unit, integration, pipeline тестов.
- **При создании PR:** дополнительно запуск SAST и сканирование зависимостей.
- **Перед выкаткой в продакшен:** обязательное выполнение E2E и нагрузочных тестов (на тестовом стенде).
- **При падении любого теста** – деплой блокируется (исключение: разрешённые пропуски для временных проблем).
- **Отчёт о покрытии кода** (`pytest-cov`) публикуется в артефактах сборки.
- **Логи тестирования** сохраняются для последующего анализа.

## 8. Критические пути (обязательное тестирование)

- Регистрация и подтверждение email.
- Аутентификация (JWT, refresh).
- Генерация промпта (успех, уточняющие вопросы, исчерпание лимитов).
- Сохранение, редактирование, удаление промпта.
- Импорт файлов (разные форматы, лимиты).
- Экспорт данных (Excel, JSON).
- Оплата подписки и пополнение счёта (вебхук + обновление баланса).
- Кэширование лимитов (инвалидация при изменении).
- Уведомления (email, Telegram) при ключевых событиях.

## 9. Особые требования для AI-пайплайна

- Мок `call_llm_chat` должен возвращать предопределённые ответы, соответствующие эталонному датасету.
- Golden dataset содержит минимум 5 кейсов на каждую модальность (text‑craft, image‑visual и т.п.).
- При изменении системных промптов (или логики пайплайна) необходимо перепрогонять golden dataset и проверять, что QC-скоры не упали ниже порога.
- Тестируется как позитивный сценарий (получение финального промпта), так и негативные (ошибка LLM, невалидный ответ, таймаут).

## 10. Метрики качества тестирования

- **Покрытие кода:** не ниже целевых значений (85% / 70%).
- **Прохождение тестов:** 100% успешных тестов перед деплоем.
- **Время выполнения тестов:** unit + integration + pipeline не более 10 минут; E2E и нагрузочные – не более 30 минут.
- **Обнаруженные дефекты:** при регрессии фиксируется новый тест, воспроизводящий ошибку.

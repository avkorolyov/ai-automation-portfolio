## 1. Цель и область применения
Настоящий документ определяет минимально необходимый и достаточный набор проектной документации Competitor AI для разработки, эксплуатации и передачи между командами.

Область применения:
- backend/frontend разработка;
- инфраструктура и эксплуатация;
- сопровождение интеграций;
- контроль соответствия реализации требованиям.

## 2. Принципы ведения документации
- Один источник истины для каждого класса информации.
- Нормативные документы отделены от рабочих черновиков.
- Документы должны быть пригодны для аудита и передачи между командами.
- Дублирование запрещено: вместо копирования использовать ссылки на первоисточник.
- Архивные и временные документы не участвуют в обязательном контуре документации.

## 3. Целевой набор документов (8+)

### 3.1 Обязательные документы
1. **`Техническое задание.md`**
   - Назначение: единый документ, формализующий функциональные и нефункциональные требования. Основная часть содержит краткое описание, детали вынесены в приложения. Документ должен содержать ссылку на `Интерфейсы и сценарии.md` для детализации интерфейсов и сценариев.
   - Аудитория: заказчик, product, аналитик, разработка, QA.

2. **`Архитектура.md`**
   - Назначение: архитектурные решения, границы систем, ключевые компоненты и интеграции.
   - Аудитория: архитектор, backend/frontend, DevOps, безопасность.

3. **`README.md`**
   - Назначение: точка входа по запуску, сборке, структуре и базовым сценариям использования.
   - Аудитория: разработка, QA, эксплуатация.

4. **`docs/Стандарты разработки кода.md`**
   - Назначение: единые правила разработки, оформления, ревью и сопровождения кода.
   - Аудитория: backend/frontend/desktop разработка, техлид, QA.

5. **`docs/Требования к автоматическому тестированию.md`**
   - Назначение: единые обязательные требования к автотестам, покрытию, тестовой безопасности и CI/CD quality gates.
   - Аудитория: backend/frontend/desktop разработка, QA, техлид, DevOps.

6. **`docs/Отчет о тестировании.md`**
   - Назначение: фиксация результатов тестирования по релизам/итерациям, выявленных дефектов и статуса их устранения.
   - Аудитория: QA, разработка, техлид, заказчик.

7. **`docs/Требования к документации на проекте.md`**
   - Назначение: правила ведения, актуализации и согласованности документации проекта.
   - Аудитория: вся команда.

8. **`docs/Интерфейсы и сценарии.md`**
   - Назначение: описание пользовательских интерфейсов и сценариев для web и desktop в едином документе с выделением общих и платформенных различий.
   - Аудитория: product, аналитик, frontend, desktop, QA.

### 3.2 Допустимые дополнительные документы
1. **`docs/docs.md`** (при необходимости)
   - Назначение: расширенная техническая документация по API и внутренним контрактам.
   - Аудитория: backend/frontend, QA.

2. **Дополнительные файлы в `docs/`** (при необходимости)
   - Назначение: стандарты, чек-листы, описания процессов, которые нужны текущему проекту.
   - Аудитория: определяется документом.

## 4. Требования к содержанию документов
### 4.1 Обязательные элементы каждого документа
Каждый документ обязательного контура (кроме шаблонов) должен содержать:
- цель документа;
- область применения;
- оглавление с якорными ссылками;
- версионность/дату актуализации;
- ссылки на связанные документы.

Для шаблонов (например, `Отчет о тестировании.md`) допускается упрощенная структура без полного оглавления при условии, что шаблон однозначно определяет порядок заполнения и состав разделов.

### 4.2 Нормативные формулировки
Для нормативных разделов применять формулировки уровня "должно быть реализовано".

### 4.3 Эксплуатационные разделы
Для эксплуатационных разделов включать проверяемые шаги и критерии результата.

### 4.4 Базовый принцип структуры для всех документов
Для всех документов обязательного контура применяется общий принцип:
- **Основная часть** — кратко фиксирует суть, цели и ключевую логику раздела;
- **Приложения (в конце документа)** — содержат детали, примеры, расширенные спецификации, чек-листы, user flows и edge cases.

### 4.5 Учет специфики документа
Состав и глубина деталей в приложениях зависят от типа документа:
- для интерфейсных документов — детальные сценарии, элементы UI, ошибки/валидации;
- для архитектурных и API-документов — детальные технические спецификации;
- для эксплуатационных документов — детальные инструкции и критерии проверки;
- для стандартов разработки — детальные правила, практики и контрольные списки.

## 5. Правила согласованности и ссылок
- Термины, обозначения, имена подсистем и интеграций должны совпадать во всех документах.
- При наличии одного первоисточника повторять информацию в других документах запрещено.
- Перекрёстные ссылки оформлять на уровень заголовков внутри документа или между файлами.
- Если документ описывает ограничения реализации, в нём должна быть ссылка на соответствующий раздел `README.md` или профильный документ в `docs/`.
- Если документ описывает требования к процессу, в нём должна быть ссылка на связанный стандарт (`Стандарты разработки кода.md`, `Требования к автоматическому тестированию.md`).

## 6. Правила жизненного цикла документов
- Рабочие черновики, планы и разовые расследования:
  - либо удаляются после закрытия задачи,
  - либо выводятся из обязательного контура документации.
- В обязательный контур документации входят файлы из `docs` и `README.md` в корне репозитория.
- Любое изменение обязательных документов должно сопровождаться сверкой:
  - с текущим кодом;
  - с `README.md`;
  - с актуальной структурой проекта (`backend`, `frontend`, `desktop`, `tests`).

## 1. Назначение и область применения
Документ задает обязательные правила оформления и проверки кода в проекте Competitor AI.

Основная часть фиксирует обязательные нормы. Практический контроль вынесен в приложение.

## 2. Единые требования ко всем языкам
- Все комментарии и docstrings в проекте пишутся **только на русском языке**.
- Комментарии описывают неочевидную логику, а не очевидные операции.
- Любое отклонение от стандартов должно быть обосновано в PR.
- Именование ключевых сущностей и путей должно соответствовать текущей структуре: `backend/*`, `frontend/*`, `desktop/*`, `tests/*`.

## 3. Стандарты Python
### 3.1 Базовые нормы
- Обязательные основы: **PEP 8** (стиль), **PEP 257** (docstrings).
- Формат docstrings: **Google Style**.
- Публичные сущности документируются: модули, классы, функции, методы.

### 3.2 Докстринги и комментарии
- Каждый Python-модуль начинается с docstring.
- Для публичных функций/методов обязательны блоки `Args`, `Returns`, `Raises` (если применимо).

Пример:
```python
def calculate_price(amount: int, discount: float = 0.0) -> float:
    """Рассчитывает итоговую цену с учетом скидки.

    Args:
        amount: Базовая сумма в рублях.
        discount: Процент скидки в диапазоне 0..100.

    Returns:
        Итоговая сумма после применения скидки.

    Raises:
        ValueError: Если скидка вне диапазона 0..100.
    """
    if not 0 <= discount <= 100:
        raise ValueError("Скидка должна быть в диапазоне 0..100")
    return amount * (1 - discount / 100)
```

### 3.3 Импорты и именование
- Порядок импортов: стандартная библиотека → сторонние пакеты → локальные пакеты проекта (`backend.*` и т.д.).
- Запрещены `from module import *` (кроме явных модулей с `__all__`) и неявные относительные импорты.
- Именование по PEP 8: `lowercase_with_underscores` для модулей/функций/переменных, `CapWords` для классов, `ALL_CAPS` для констант.

### 3.4 Типизация
- Типизация обязательна для публичных функций/методов и значимой внутренней логики.
- `Any` запрещен; исключение допускается только с явным комментарием-обоснованием.
- API-контракты и валидация: Pydantic.
- Для локальной валидации допускается `py_compile` и линтеры IDE; расширенные проверки (`mypy`) применяются по необходимости.

## 4. Стандарты Web (HTML/CSS/JS)
### 4.1 JavaScript
- Использовать понятные имена функций и переменных, избегать скрытых побочных эффектов.
- Новые функции должны быть небольшими и решать одну задачу.
- Для неочевидных веток и преобразований допускаются короткие поясняющие комментарии.

### 4.2 DOM и UI
- Идентификаторы элементов (`id`) должны быть стабильными и соответствовать назначению.
- Стили и скрипты для одной фичи синхронизируются: изменение разметки должно сопровождаться проверкой `app.js` и `styles.css`.
- Любые изменения пользовательских сценариев (вкладки, модальные окна, загрузка файлов) должны быть проверены вручную в UI.

### 4.3 Ошибки и обработка запросов
- Ошибки в асинхронных операциях обрабатываются через `try/catch`.
- Для каждого API-вызова нужно явно обрабатывать неуспешный ответ и показывать понятное сообщение пользователю.
- Новые endpoint-ы должны быть связаны с соответствующим отображением в истории запросов.

## 5. Инструменты автоматического контроля
Обязательные проверки в CI:
- Python: линтеры/проверки проекта по текущей конфигурации репозитория;
- Frontend: ручная функциональная проверка сценариев после изменений;
- Smoke-проверка API: `tests/smoke_check.py` по необходимости.

Рекомендуемая база:
- Python: единый стиль PEP 8, читаемая типизация, понятные сообщения ошибок;
- Web: единый стиль в `frontend/styles.css`, отсутствие дублирующей логики в `frontend/app.js`.

Нарушение обязательных проверок блокирует слияние PR.

## 6. Минимальные требования безопасности в коде
- Не хардкодить секреты, токены, пароли, ключи.
- Запрещено `except Exception: pass`.
- Не принимать пользовательские файлы без проверок типа/размера.
- При работе с файлами использовать безопасные имена и ограничивать директории записи.

## 1. Цель документа
Определить пошаговый план внедрения и развития автотестов для Competitor AI так, чтобы обеспечить стабильное покрытие кода выше 80% и контроль критических пользовательских сценариев.

## 2. Область применения
План распространяется на:
- backend (`backend/*`);
- desktop-клиент (`desktop/*`);
- frontend-клиентскую логику (`frontend/app.js`);
- smoke-контур (`tests/smoke_check.py`, `tests/data/*`).

## 3. Исходные условия
- Критические сценарии уже описаны в `docs/Отчет о тестировании.md`.
- В проекте есть рабочий smoke-контур (`tests/smoke_check.py`).
- Документация и код приведены к единым стандартам.
- Тесты внедряются инкрементально, приоритет — критический пользовательский поток.

## 4. Целевые показатели
- Суммарное покрытие автотестами Python-кода проекта: **>80%**.
- Критические endpoint-ы (`/analyze/text`, `/analyze/image`, `/analyze/pdf`, `/parse/demo`, `/history`) покрыты позитивными и негативными сценариями.
- Все релизные проверки проходят без `BLOCKER`/`CRITICAL` дефектов.

## 5. Стратегия внедрения
- Сначала покрываются unit-тестами сервисы и утилиты с высокой ветвистостью.
- Затем покрываются интеграционно endpoint-ы FastAPI.
- После стабилизации backend добавляются проверки frontend/desktop клиентской логики.
- `smoke_check` используется как обязательный первый этап общего тест-пайплайна.
- Полный запуск выполняется единым пайплайном: `python tests/run_quality_checks.py`.

## 6. План по итерациям

### Итерация 1 — Базовый backend-контур
**Цель:** закрыть основные endpoint-ы и критичные ошибки API.

Сценарии:
- `POST /analyze/text`: успех, валидация, ошибка LLM.
- `POST /analyze/image`: успех, пустой файл, ошибка LLM, запись uploads/history.
- `POST /analyze/pdf`: успех, пустой файл, невалидный PDF, пустой текст, ошибка LLM.
- `GET /history`, `DELETE /history`: корректность ответов.

Ожидаемый вклад в покрытие:
- +25..35% к покрытию Python-кода.

### Итерация 2 — Сервисный слой и файловые эффекты
**Цель:** стабилизировать поведение `history_service`, `parsing_service`, `llm_service`.

Сценарии:
- Миграция формата истории, лимит `max_history_items`.
- Selenium fallback и повторный проход при коротком контенте.
- Нормализация JSON-ответов LLM (`_extract_json`, `_to_str_list`, `_normalize_competition_analysis`).
- Проверка очистки runtime-файлов (`uploads`, `screenshots`, `tmp_uploaded.pdf`).

Ожидаемый вклад в покрытие:
- +20..25%.

### Итерация 3 — Frontend/desktop клиентская логика
**Цель:** зафиксировать стабильное поведение UI-слоя при API-ошибках и валидациях.

Сценарии:
- Frontend: обработка `res.ok === false` для всех API-вызовов.
- Frontend: ограничения по размеру/типу (text/image/pdf).
- Desktop: `APIClient` (ConnectionError/Timeout/HTTPError), `health`, базовые настройки URL.

Ожидаемый вклад в покрытие:
- +10..15% (Python) + контроль клиентских сценариев.

### Итерация 4 — Регрессия и релизный контур
**Цель:** сделать прогоны воспроизводимыми и релизно-ориентированными.

Сценарии:
- Полный прогон общего пайплайна `tests/run_quality_checks.py`.
- Этап 1: `tests/smoke_check.py` (базовая стабильность API).
- Этап 2: `tests/fullscale_check.py` (расширенные позитивные/негативные и runtime-сценарии).
- Ретест критичных дефектов.
- Проверка критериев из `docs/Отчет о тестировании.md`.

Ожидаемый результат:
- Стабильное покрытие Python-кода >80%.
- Повторяемые релизные прогоны без ручной донастройки.

## 7. Критерии готовности (Definition of Done)
- Все сценарии текущей итерации имеют статусы `PASS` или `N/A` с обоснованием.
- Изменения в API сопровождаются актуализацией тестов и документации.
- Покрытие не падает ниже согласованного порога.
- Отчет о тестировании заполнен по фактическим результатам.

## 8. Риски и меры снижения
- Нестабильность внешних зависимостей (LLM/сеть) -> использовать моки в unit/integration уровнях.
- Хрупкость парсинга сайтов -> фиксировать стабильные тестовые URL и сценарии fallback.
- Регрессии в UI-валидации -> включить отдельные клиентские проверки в каждый релизный прогон.

## 9. Формат отчетности
- Результаты каждого этапа фиксируются в `docs/Отчет о тестировании.md`.
- Для каждой итерации указываются:
  - выполненные сценарии;
  - покрытие (строки/ветви);
  - список дефектов;
  - решение `Go/No-Go`.

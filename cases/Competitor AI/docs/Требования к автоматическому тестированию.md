## 1. Назначение и область применения
Документ задает обязательные требования к автоматическому тестированию в проекте Competitor AI и фиксирует минимальный проверяемый контур перед релизом.

Требования распространяются на:
- backend API;
- web-интерфейс (`frontend`);
- desktop-клиент (`desktop`) как оболочку над backend API;
- smoke-проверки в `tests/smoke_check.py`.

## 2. Общие принципы
1. Автотесты проверяют реализованный функционал проекта и не должны описывать несуществующие подсистемы.
2. Тестовые данные хранятся отдельно от runtime-данных приложения: используется `tests/data`.
3. Критические пользовательские сценарии проверяются автоматически в рамках smoke-контура.
4. В тестах не допускается использование реальных секретов и production-ключей.
5. При изменении API-контрактов, структуры ответов или пользовательских сценариев тесты актуализируются в том же изменении.

## 3. Обязательный контур тестирования

### 3.1 Общий тест-пайплайн (обязательный)
Обязательный запуск перед релизом: `python tests/run_quality_checks.py`.

Состав общего пайплайна:
- Этап 1: `python tests/smoke_check.py`.
- Этап 2: `python tests/fullscale_check.py`.
- Этап 3: `python tests/contract_api_check.py`.
- Этап 4: `python tests/resilience_check.py`.
- Этап 5: `python tests/security_check.py`.
- Этап 6: `python tests/e2e_ui_check.py`.

`smoke_check` является частью общего пайплайна и не запускается как единственная релизная проверка.

Отдельный обязательный этап для закрытия quality gate по покрытию:
- `python tests/run_coverage_checks.py`.
- Этап включает:
  - backend coverage через `coverage.py`;
  - frontend coverage через `vitest --coverage` для `frontend/app.js`.

### 3.2 API smoke (обязательный этап общего пайплайна)
`smoke_check` должен покрывать базовые endpoint-ы:

Smoke-проверка должна покрывать:
- `POST /analyze/text`;
- `POST /analyze/image`;
- `POST /analyze/pdf`;
- `POST /parse/demo`;
- `GET /history`;
- `DELETE /history`.

### 3.3 Проверки по сценариям
1. **Анализ текста:** успешная обработка валидного текста и корректная структура ответа.
2. **Анализ изображений:** обработка всех доступных тестовых изображений из `tests/data/images`.
3. **Анализ PDF:** обработка всех PDF из `tests/data/pdfs` и корректная реакция на невалидные данные.
4. **Парсинг URL:** запуск сценария для всех URL из `tests/data/urls.txt`.
5. **История:** запись результатов сценариев в историю и корректная очистка через `DELETE /history`.

### 3.4 Проверка очистки runtime-данных
После вызова `DELETE /history` должно быть подтверждено:
- очищение истории (`data/history.json`);
- удаление runtime-файлов из `data/uploads` и `data/screenshots`;
- удаление `data/tmp_uploaded.pdf` при наличии.

## 4. Данные и среда тестирования

### 4.1 Структура тестовых данных
Используется следующая структура:

```text
tests/
├── run_quality_checks.py
├── smoke_check.py
├── fullscale_check.py
└── data/
    ├── texts/
    ├── images/
    ├── pdfs/
    └── urls.txt
```

Для расширенного контура используются дополнительные данные:

```text
tests/data/fullscale/
├── texts/
├── images/
├── pdfs/
├── urls/
└── history/
```

### 4.2 Среда запуска
- Python-окружение проекта (`venv`).
- Установленные зависимости из `requirements.txt`.
- Запущенный backend (`python backend/run.py`) перед smoke-запуском.

## 5. Нефункциональные требования к тестам
- Тесты должны быть воспроизводимыми и не зависеть от времени суток.
- Имена проверок и сообщений об ошибках должны быть понятными и указывать на конкретный сценарий.
- Ошибки тестов должны быть диагностируемыми без чтения исходного кода теста.
- Время smoke-прогона должно оставаться приемлемым для регулярного запуска командой.

## 6. Минимальные quality gates
- Перед релизной сборкой desktop обязателен успешный прогон `tests/run_quality_checks.py`.
- Перед релизом обязателен успешный прогон `tests/run_coverage_checks.py` с порогом покрытия backend Python-кода `>=80%`.
- При падении любого этапа пайплайна (`smoke_check` или `fullscale_check`) релиз блокируется до устранения причины.
- Изменения, затрагивающие API/интерфейсы/сборку, должны сопровождаться актуализацией документации (`README.md` и `docs/*`).

Примечание по области покрытия:
- coverage в данном проекте измеряет Python-код backend (`backend/*`);
- `frontend/*` (JavaScript/CSS/HTML) не входит в Python coverage и измеряется отдельным JS-контуром.

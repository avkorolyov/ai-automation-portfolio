## 1. Цель документа
Описать пользовательские интерфейсы и ключевые сценарии работы системы в едином формате для web и desktop.

## 2. Область применения
Документ применяется product/аналитиком, frontend и desktop разработчиками, а также QA для единого описания пользовательских сценариев и критериев приемки интерфейсов.

## 3. Подход к описанию
- **Общие сценарии** фиксируются один раз.
- **Платформенные отличия** (web/desktop) выделяются отдельными подразделами.
- Дублирование одинаковой логики между web и desktop не допускается.

## 4. Карта интерфейсов

### 3.1 Основные разделы (общие)
- Анализ текста
- Анализ изображений
- Анализ PDF
- Парсинг сайта
- История запросов
- Модальное окно просмотра деталей запроса

### 3.2 Desktop-специфика
- Боковое desktop-меню-оболочка (кнопки разделов).
- Экран и модальное окно настроек backend URL.
- Индикатор статуса соединения с backend.

### 3.3 Web-специфика
- Отображение разделов в веб-странице через tab-панели.
- Загрузка изображений/PDF через drag-and-drop.

## 5. Общие сценарии пользователя

### 4.1 Анализ текста
1. Пользователь открывает вкладку "Анализ текста".
2. Вводит текст конкурента.
3. Нажимает "Проанализировать".
4. Система отправляет запрос в `POST /analyze/text`.
5. Отображается структурированный результат:
   - сильные стороны;
   - слабые стороны;
   - уникальные предложения;
   - рекомендации;
   - резюме.

### 4.2 Анализ изображений
1. Пользователь открывает вкладку "Анализ изображений".
2. Загружает файл (drag-and-drop или выбор из файловой системы).
3. Клиент валидирует входные ограничения:
   - формат: PNG/JPG/JPEG/GIF/WEBP;
   - размер файла: не более **10 MB**.
4. Нажимает "Проанализировать".
5. Система отправляет запрос в `POST /analyze/image`.
6. Отображается структурированный результат:
   - краткое описание изображения;
   - ключевые визуальные инсайты;
   - оценка визуального стиля (score);
   - анализ стиля и подачи;
   - рекомендации по улучшению.

### 4.3 Анализ PDF
1. Пользователь открывает вкладку "Анализ PDF".
2. Загружает PDF-файл (drag-and-drop или выбор из файловой системы).
3. Клиент валидирует входные ограничения:
   - формат: PDF;
   - размер файла: не более **20 MB**.
4. Нажимает "Проанализировать PDF".
5. Система отправляет запрос в `POST /analyze/pdf`.
6. Отображается результат в формате текстового анализа.

### 4.4 Парсинг сайта
1. Пользователь открывает вкладку "Парсинг сайта".
2. Вводит URL.
3. Нажимает "Запустить парсинг".
4. Система отправляет запрос в `POST /parse/demo`.
5. Отображаются:
   - карточка страницы (URL/Title/H1/первый абзац/число чанков);
   - структурированный аналитический результат:
     - сильные стороны;
     - слабые стороны;
     - уникальные предложения;
     - рекомендации;
     - резюме.

### 4.5 История запросов
1. Пользователь открывает вкладку "История".
2. Система загружает данные из `GET /history`.
3. Пользователь выбирает карточку истории.
4. Открывается модальное окно с двумя секциями:
   - запрос;
   - результат.
5. При нажатии "Очистить историю" вызывается `DELETE /history`.

## 6. Состояния и обработка ошибок

### 5.1 Состояния загрузки
- Во время выполнения запроса кнопка переводится в loading-state.
- Повторные клики по кнопке блокируются до завершения запроса.

### 5.2 Пустые/невалидные входы
- Текст: минимальная длина контролируется на клиенте (**не менее 20 символов**).
- Изображение: проверяется факт выбора и лимит размера (**не более 10 MB**).
- PDF: проверяется факт выбора, формат и лимит размера (**не более 20 MB**).
- URL: передается в backend для валидации и парсинга.

### 5.3 Отображение ошибок
- Ошибки backend и валидации отображаются в карточке результатов.
- Для истории при ошибке загрузки показывается fallback-сообщение.

## 7. Различия Web и Desktop

### 6.1 Что совпадает
- Бизнес-сценарии и API-вызовы полностью совпадают.
- Набор вкладок и выходные данные анализа совпадают.
- История и модальное окно деталей работают одинаково.

### 6.2 Что отличается
- Desktop имеет отдельную оболочку приложения и меню.
- Desktop содержит UI настроек backend URL и проверку статуса подключения.
- Web работает как самостоятельная страница в браузере.

## 8. Критерии приемки интерфейсов
- Все 5 основных сценариев (текст/изображение/PDF/парсинг/история) выполняются без ошибок.
- Для каждого сценария отображается корректный результат или понятная ошибка.
- В desktop настраивается backend URL и отображается статус подключения.
- Очистка истории удаляет записи и связанные runtime-файлы.

## 1. Цель документа
Описать архитектурные решения проекта Competitor AI: границы системы, ключевые компоненты, взаимодействия и контур сборки desktop-приложения.

## 2. Область применения
Документ применяется командами backend/frontend/desktop разработки, DevOps и QA для согласования технической структуры и ключевых архитектурных ограничений.

## 5. Ключевые runtime-потоки

### 5.1 Sequence: анализ изображения

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant UI as Web/Desktop UI
    participant API as backend/main.py
    participant LLM as llm_service
    participant FS as data/uploads
    participant H as history_service

    U->>UI: Загружает изображение и запускает анализ
    UI->>API: POST /analyze/image (multipart)
    API->>LLM: analyze_image(...)
    LLM-->>API: JSON-результат анализа
    API->>FS: Сохранение исходного файла
    API->>H: add("analyze_image", input/result)
    API-->>UI: HTTP 200 + JSON
    UI-->>U: Показ результата
```

### 5.2 Sequence: парсинг сайта

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant UI as Web/Desktop UI
    participant API as backend/main.py
    participant P as parsing_service
    participant S as Selenium/requests
    participant LLM as llm_service
    participant H as history_service
    participant FS as data/screenshots

    U->>UI: Ввод URL и запуск парсинга
    UI->>API: POST /parse/demo
    API->>P: parse_and_analyze(url)
    P->>S: parse_with_selenium(url)
    alt Selenium недоступен/ошибка
        P->>S: parse_fallback(url)
    end
    S-->>P: title, text, h1, first_paragraph
    P->>FS: save_screenshot(parse_demo.png) (если Selenium успешен)
    alt Извлеченный текст слишком короткий (< 1000 символов)
        P->>S: parse_with_selenium(url, extended wait/scroll)
        S-->>P: обновленные title, text, h1, first_paragraph
    end
    P->>LLM: analyze_parsed_content(...)
    LLM-->>P: strengths/weaknesses/...
    P-->>API: итоговый payload
    API->>H: add("parse_demo", input/result)
    API-->>UI: HTTP 200 + JSON
    UI-->>U: Показ результата парсинга и аналитики
```

### 5.3 Sequence: очистка истории

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant UI as Web/Desktop UI
    participant API as backend/main.py
    participant H as history_service
    participant Hist as data/history.json
    participant FS as data/uploads + data/screenshots + data/tmp_uploaded.pdf

    U->>UI: Нажимает "Очистить историю"
    UI->>API: DELETE /history
    API->>H: clear()
    H->>Hist: write {"version": 2, "items": []}
    API->>FS: Очистка uploads/screenshots/tmp_uploaded.pdf
    API-->>UI: {"status":"cleared"}
    UI-->>U: Показ статуса "История очищена"
```

## 8. Принятые архитектурные решения
- **Разделение по слоям:** `backend`, `frontend`, `desktop`, `tests`.
- **Единый backend-контур:** web и desktop используют одинаковые API-эндпоинты.
- **File-based runtime storage:** история и загруженные файлы хранятся локально в `data/`.
- **Устойчивость парсинга:** Selenium-first с fallback на requests/BeautifulSoup.
- **Явная очистка runtime-данных:** `DELETE /history` очищает и историю, и временные файлы.

## 9. Ограничения архитектуры
- Нет выделенной БД; история хранится в JSON-файле.
- Desktop зависит от доступности backend-процесса.
- Качество ответов зависит от внешнего LLM-провайдера.
- Кросс-компиляция desktop-сборок не поддерживается (сборка по целевой ОС).

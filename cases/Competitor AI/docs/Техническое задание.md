## 1. Цель документа
Зафиксировать функциональные и нефункциональные требования проекта Competitor AI, а также ограничения и правила эксплуатации.

## Область применения
Документ применяется для согласования требований между заказчиком, аналитиком, разработкой и QA и описывает реализованный функционал текущей версии проекта.

## 2. Назначение системы
Competitor AI - мультимодальный ассистент для анализа конкурентных материалов:
- текстов;
- изображений;
- PDF-документов;
- контента веб-страниц по URL.

Система предоставляет web-интерфейс, desktop-клиент и HTTP API.

## 3. Границы системы

### 3.1 Что входит в проект
- Backend API на FastAPI (`backend`).
- Web UI на HTML/CSS/JS (`frontend`).
- Desktop GUI на PyQt6 (`desktop`), работающий поверх backend API.
- Тестовый smoke-контур (`tests/smoke_check.py` + `tests/data`).
- Скрипт сборки desktop-релизов (`desktop/build.py`).

### 3.2 Что не входит
- Авторизация пользователей и ролевая модель.
- Платежи и биллинг.
- Управление аккаунтами/админ-панель.

## 4. Функциональные требования

### 4.1 API
Система должна предоставлять endpoint-ы:
- `GET /health` - проверка доступности сервиса.
- `POST /analyze/text` - анализ текста конкурента.
- `POST /analyze/image` - анализ изображения конкурента.
- `POST /analyze/pdf` - анализ PDF.
- `POST /parse/demo` - парсинг и анализ сайта по URL.
- `GET /history` - получение истории запросов.
- `DELETE /history` - очистка истории и runtime-файлов.

### 4.2 Анализ текста
- Вход: `competitor_name`, `text`.
- Ограничение: текст не короче минимального порога валидации.
- Выход (структурированно): `strengths`, `weaknesses`, `unique_offers`, `recommendations`, `summary`.

### 4.3 Анализ изображений
- Вход: multipart с файлом изображения.
- Поддерживаемые типы на UI: PNG/JPG/JPEG/GIF/WEBP.
- Файл сохраняется в `data/uploads`, ссылка на файл сохраняется в истории.
- Выход: описание, инсайты, скор визуального стиля, анализ стиля, рекомендации.

### 4.4 Анализ PDF
- Вход: multipart с файлом PDF.
- Текст извлекается через `pypdf`.
- При невалидном/пустом PDF сервис возвращает ошибку валидации.
- Выход соответствует формату текстового анализа.

### 4.5 Парсинг сайта
- Вход: URL.
- Основной режим: Selenium + анализ DOM.
- Fallback: `requests` + BeautifulSoup.
- Выход: метаданные страницы (`title`, `h1`, `first_paragraph`, `analyzed_chunks`) и результат анализа.

### 4.6 История и очистка данных
- История хранится в `data/history.json`.
- `DELETE /history` должен:
  - очищать историю;
  - очищать runtime-файлы в `data/uploads` и `data/screenshots`;
  - удалять временный `data/tmp_uploaded.pdf`, если существует.

### 4.7 Web UI
- Web UI должен предоставлять вкладки:
  - анализ текста;
  - анализ изображений;
  - анализ PDF;
  - парсинг;
  - история.
- Для изображений и PDF должна быть загрузка через drag-and-drop/выбор файла.
- История должна открывать модальное окно с исходным запросом и результатом.

### 4.8 Desktop UI
- Desktop-клиент должен отображать те же ключевые сценарии, что и web UI.
- Desktop-клиент должен иметь экран настроек backend URL и статус соединения.
- Desktop-клиент работает как GUI-оболочка и обращается к backend API.

## 5. Нефункциональные требования

### 5.1 Производительность
- API должен отвечать в приемлемое время для пользовательских сценариев.
- Длительные операции (парсинг/LLM-анализ) должны корректно обрабатывать таймауты и ошибки.

### 5.2 Надежность
- Ошибки внешних зависимостей (LLM, Selenium, сеть) не должны приводить к падению приложения.
- API должен возвращать информативные HTTP-ошибки.

### 5.3 Безопасность
- Секреты хранятся только в `.env`.
- Имена загружаемых файлов должны санитизироваться.
- Runtime-данные очищаются через endpoint очистки истории.

### 5.4 Поддерживаемость
- Проектная структура разделена на `backend`, `frontend`, `desktop`, `tests`.
- Документация и требования должны поддерживаться в актуальном состоянии при изменении кода.

## 6. Технологические требования
- Python 3.13+
- FastAPI + Uvicorn
- PyQt6 (desktop)
- Selenium + BeautifulSoup
- OpenAI SDK через Polza.ai gateway
- PyInstaller (сборка desktop)

## 7. Сборка и релизы
- Скрипт сборки: `desktop/build.py`.
- Основные профили:
  - `macos` -> `dist/Competitor AI.app`
  - `windows` -> `dist/Competitor AI.exe`
- Папки `build`, `dist`, `release` создаются автоматически при сборке.

## 8. Тестирование и приемка
- Базовая автоматическая проверка: `tests/smoke_check.py`.
- Датасет smoke-проверок: `tests/data`.
- Критерий приемки: успешный проход smoke-check и ручная проверка ключевых UI-сценариев.

## 9. Ограничения и допущения
- Кросс-компиляция PyInstaller не поддерживается: профиль собирается на целевой ОС.
- Качество аналитики зависит от внешней LLM и доступности сетевых интеграций.

# Competitor AI — публичный пакет кейса

> **Статус:** демонстрационный MVP, не завершенный промышленный продукт.  
> Решение предназначено для демонстрации уровня проектирования, разработки, документирования, качества кода и тестирования. Перед внедрением в production необходимо уточнение ТЗ под контур клиента, доработка архитектуры/интеграций и полный цикл приемочного тестирования.

Мультимодальный AI-сервис для анализа конкурентных материалов (текст, изображения, PDF, сайты) с единым API-контуром, web UI и desktop-оболочкой.

## 1. Контекст задачи
Во многих командах анализ конкурентной среды выполняется вручную: маркетинговые материалы, скриншоты лендингов, презентации в PDF и контент сайтов обрабатываются разрозненно и без единых стандартов.

Это приводит к:
- высокой операционной нагрузке и повторяющимся ручным действиям;
- нестабильному качеству выводов между разными аналитиками;
- сложностям масштабирования процесса без роста штата;
- слабой интеграции AI в существующие бизнес-процессы.

Задача кейса — показать, как этот контур формализуется в инженерное решение с прозрачными контрактами API, fallback-устойчивостью и проверяемым качеством.

## 2. Логика решения
Вместо набора отдельных утилит реализован единый pipeline конкурентного анализа:
- прием разных типов входных данных (text/image/pdf/url);
- унификация обработки через API-first backend;
- сохранение результатов и истории в стандартной схеме;
- fallback-обработка ошибок внешних AI/парсинг-зависимостей;
- контроль качества через единый тестовый контур и coverage-gate.

Подход ориентирован на интеграцию AI в работающую инфраструктуру, а не на изолированные demo-сценарии.

## 3. Архитектура решения
Решение реализовано как модульная система:
- backend (`FastAPI`) с endpoint-ами анализа и историей;
- frontend (HTML/CSS/JS) для web-сценария;
- desktop GUI (PyQt6), работающий поверх backend API;
- тестовый контур (smoke/fullscale/contract/resilience/security/e2e);
- набор нормативных и проектных документов.

Ключевые архитектурные принципы:
- API-first для всех клиентских каналов;
- fallback-first при внешних сбоях;
- разделение на контуры: runtime-контур (`backend`, `frontend`, `desktop`), контур качества (`tests`) и контур документационного сопровождения (`docs`) для понимания логики решения, тестирования и эксплуатации.

## 4. Реализация
В рамках кейса реализованы:
- анализ текста (`POST /analyze/text`);
- анализ изображений (`POST /analyze/image`);
- анализ PDF (`POST /analyze/pdf`);
- парсинг и анализ сайтов (`POST /parse/demo`);
- управление историей (`GET/DELETE /history`);
- автотесты с формальной проверкой покрытия backend + frontend.

Подробные технические инструкции по установке, запуску, тестированию и сборке вынесены в отдельный документ:  
[`docs/Руководство по установке и запуску.md`](./docs/Руководство%20по%20установке%20и%20запуску.md)

## 5. Результат
Кейс демонстрирует:
- сокращение ручных операций в задачах конкурентной аналитики;
- стабильность обработки за счет fallback-логики;
- управляемость качества через документацию и автотесты;
- готовность к адаптации под контуры конкретного заказчика.

## 6. Уровень участия
Выполнены ключевые проектные и инженерные работы:
- анализ и формализация требований;
- проектирование архитектуры и API-контрактов;
- реализация backend/web/desktop контуров;
- внедрение тестовой стратегии и контроль покрытия;
- подготовка полного комплекта проектной документации.
